Background

Traditionally, data mining was done using a suite of different tools:

  • Diagramming tools (such as MS Visio) for system specification.
  • ETL tools (such as Pentaho Kettle, now part of Hitachi's DataOps suite) for acquiring data, often from operational databases and sometimes from data warehouses.
  • SQL scripts for combining and transforming the data en route (e.g., generating statistical summaries).
  • Proprietary statistical packages (such as SAS, SPSS (now acquired by IBM), etc.) for the more complex analyses and visualisations.
  • A mixture of languages and technologies (shell scripts, Java, C++, Javascript, etc.) to build the overall application that is deployed in production, e.g., to provide a web application for fraud checking.

Nowadays we distinguish between two data mining usage patterns:

  • exploring the data and hence the underlying data-generating processes, and relating this data to a business/research objective, by analysts,
  • adding value by deploying the analysis procedures thus discovered to data in production, and integrating it with business processes, by software engineers.

Traditional approaches introduce a lot of complexity (in the form of interfaces and context shifts) and require significant know-how, and are not optimal for either usage pattern.

Many analysts like to use a single, GUI-based tool for their workflows. ISL Clementine (spun out from the University of Birmingham in the 1990s) was the tool that introduced the workflow metaphor and is now available as IBM SPSS Modeler Professional. Such tools are expensive, which has encouraged competition from lower price/free equivalents. KNIME is one of its (commercial) open source competitors (others include RapidMiner and orange), using the same template as the original product but with enhanced features and performance.

More recently, the R (and python) data science communities have embraced the literate programming paradigm of Knitr and Jupyter notebooks which allows them to script function calls to powerful libraries, weaving this code with its own output (including graphs) and text that is typeset as LaTeX or markdown, respectively. Such code can be extracted and inserted into applications using Big Data frameworks with relatively little effort.

Both the R and python data science communities are very active and solutions such as KNIME now position themselves as platforms that can be extended with user-provided modules written in a variety of languages, notably including R and python.

Therefore, we will use Jupyter as the primary learning tool, both for sourcing the data and for the data mining process itself.

Installation

While any Python 3.11 distribution should be suitable for our needs, we recommend you either install a full anaconda distribution (about 3.1 GB) or a reduced miniconda distribution (about 248MB).

Which should you choose? To be honest, it doesn't really matter but if you want to:

  • get python development up and running quickly while suffering poor connection issues — use miniconda.
  • get more experience installing packages — use miniconda.
  • avoid (as much as possible) all that package management "stuff" — use anaconda.
  • play around and explore python tools — use anaconda.

So a miniconda user will have faster initial download and less hard disk space used, but will need to install extra packages more frequently than an anaconda user.

It is possible to have multiple python distributions installed on your system — you just have to manage the system paths so to allow you to switch between.

Finally, there are other options — all of which should be suitable for this course's needs. However, while we are happy to help sort out issues our priority will be sorting issues in anaconda and miniconda.

  • Standard Python from Python Software Foundation.
  • On OS X using homebrew of MacPorts, see Thoughts on macOS Package Managers.
  • Enthought Deployment Manager (EDM).

Anaconda (Full python distribution)

To install anaconda:

  • Pick any suitable 64-bit, Python 3.11 installer for you system from anaconda.
  • Run installer and default options should work for your system (But to be sure, read options and verify that any option like "Add python to PATH environment variable", "Increase length of PATH environment variable", or "Register Anaconda as my default Python" is enabled). Under MacOS, it is customary to install into the home directory, so will end up with a folder ~/anaconda3.

Miniconda (python distribution)

To install miniconda:

  • Pick any suitable 64-bit, Python 3.11 installer for your system from miniconda.
  • Run installer and default options should work for your system. Under OS X, a typical install will end up with a folder ~/miniconda3.

We will assume that people who select the miniconda know what they are doing and so the the rest of this guide is more focused on Anaconda not Miniconda.

Verify Installation

Regardless of anaconda/miniconda option you picked, once installed you should be able to run python from the command line.

python

to get output and a python prompt similar to below (The important bit is "Python 3.11".)

Python 3.11.5 (main, Sep 11 2023, 08:31:25) [Clang 14.0.6 ] on darwin
Type "help", "copyright", "credits" or "license" for more information.
>>>

Type 1+1 at the prompt and press enter. You should get 2 or we are in trouble(!).

>>> 1+1
2
>>> quit()

Use quit() to exit the python shell and return to the command prompt.

Extra Packages

Both anaconda and miniconda use a package manager called conda. To install a package using anaconda, you can use the GUI or the command line as shown below. Miniconda does not offer the option of using a GUI.

conda install -y PACKAGE_NAME

The -y option causes the install to automatically carry out required installs/updated instead of asking for user confirmation.

Note: some packages are not installable using conda, in these cases or if you are using a standard (non-anaconda variant) distribution you can install using the pip installer using 

pip install PACKAGE_NAME

Side Note: Environments

Both conda and pip support the idea of environments where within the one python distributions you might have a multiple of environments. This is to help avoid version conflicts between modules. anaconda and miniconda both start with one default environment called base. If you wish to use environments, you need to apply two steps: 1) create an environment using 

conda create -n data_mining

and 2) activate it using 

conda activate data_mining

Since all of the packages that we plan to use work are (currently, Jan 2024) compatible, we won't need to use multiple environments, but they are a good idea and if you are interested there is a nice introduction at Happy Belly Bioinformatic.
